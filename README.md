# Лабораторная работа №1. Источники задач и контракты

* **Цель:** Освоить duck typing и контрактное программирование на примере источников задач, реализовав подсистему приёма задач, способную работать с различными источниками через единый поведенческий контракт без использования наследования.

* **Библиотеки:** `pytest`, `unittest.mock`, `typing`, `random`, `time`, `dataclasses`

* **Допущения:**
  - Задача представляет собой структуру с двумя полями: `id` (уникальный идентификатор, строка) и `payload` (произвольные данные, строка).
  - В качестве представления задачи используется `dataclass` (`src.classes.task_dataclass.Task`).
  - Все источники задач реализуют единый контракт — метод `get_tasks() -> list[Task]`.
  - Контракт описан с помощью `typing.Protocol` и декоратора `@runtime_checkable`.
  - Запрещено использовать общий базовый класс для источников; новые источники можно добавлять без изменения существующего кода (достаточно реализовать протокол).
  - Проверка соблюдения контракта выполняется функцией `receive` через `isinstance(source, TaskSource)`.

* **Чему я научился:**
  - Использовать `Protocol` из модуля `typing` для описания поведенческого контракта.
  - Применять декоратор `@runtime_checkable` для поддержки `isinstance` во время выполнения.
  - Реализовывать архитектуру, позволяющую легко добавлять новые источники (файл, генератор, API‑заглушка) без изменения существующего кода.
  - Писать модульные тесты с `pytest` и моками для проверки корректности работы источников и протокола.
  - Использовать аннотации типов для всех публичных интерфейсов.
  - Строить простое консольное меню для демонстрации работы.

### Алгоритм решения
1. Определён протокол `TaskSource` (в `src.classes.task_source_protocol`) с единственным методом `get_tasks()`. Протокол помечен `@runtime_checkable`, что позволяет проверять его через `isinstance`.
2. Создан простой класс `Task` (в `src.classes.task_dataclass`) с использованием `dataclass` для представления задачи.
3. Реализованы три источника, каждый из которых **не** наследует общий базовый класс, но неявно следует протоколу `TaskSource`:
   - `FileTaskSource` — читает задачи из текстового файла; каждая строка содержит `id` и `payload`, разделённые пробелом.
   - `TaskGenerator` — генерирует случайное количество задач со случайными `id` и предопределёнными вариантами `payload`.
   - `ApiTaskSource` — имитирует внешний API: может задерживать ответ (`time.sleep`) и возвращает задачи со сгенерированными `id` вида `api_*`.
4. Функция `receive` (в `src.recieve_tasks`) принимает любой объект, проверяет его соответствие протоколу (`isinstance(source, TaskSource)`) и вызывает `source.get_tasks()`. Если объект не реализует протокол, выбрасывается `TypeError`.
5. В `main.py` реализовано консольное меню, позволяющее выбрать источник, получить задачи и просмотреть их.
6. Для каждого источника и для функции `receive` написаны тесты (папка `tests/`), проверяющие:
   - корректную работу источников;
   - соблюдение протокола;
   - обработку исключений (например, отсутствие файла);
   - имитацию задержки в API‑заглушке.

### Интересные тест-кейсы
- Объект без метода `get_tasks` (`BadSource`) не проходит проверку `isinstance` на протокол.
- Чтение из несуществующего файла — ожидаемое исключение `FileTaskSourceNotFound`.
- Файл с некорректными строками (менее двух слов) — такие строки игнорируются, остальные задачи загружаются.
- Генератор создаёт задачи с `id` в диапазоне от 1 до `tasks_number*10`.
- API‑заглушка при включённой имитации задержки действительно вызывает `time.sleep`.
- Проверка, что `receive` пробрасывает исключения из источника (например, `FileTaskSourceNotFound`), а не перехватывает их.

### Инструкция к использованию

#### Запуск приложения
```bash
python -m src.main
```

### Меню

- Прочитать задачу из файла – запрашивает имя файла. Формат файла: каждая строка содержит <id> <payload> (разделитель — пробел). Пустые строки и строки с менее чем двумя словами игнорируются.

- Сгенерировать задачи – запрашивает количество, создаёт задачи со случайными данными.

- API заглушка – запрашивает количество, имитирует сетевую задержку и возвращает задачи.

- Увидеть задачи – выводит все накопленные задачи (в формате id: payload).

- Выход – завершение программы.

```

4 - Увидеть задачи

Выберите пункт: 4
1: Do homework

Выберите пункт: 5
```
